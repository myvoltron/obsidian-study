# 1. 윈도우 시스템 부팅 프로세스
## 1.1 부팅 프로세스
윈도우 시스템의 부팅은 전원이 인가된 이후 펌웨어, 부트 코드, 운영체제 이미지, 커널, 초기화 프로세스, 로그인 화면으로 이어지는 단계적 흐름이다.
### 1.1.1 부팅 단계
1. **전원 인가**
    - 컴퓨터에 전원이 들어온다.
    - CPU가 0번지 데이터를 읽는다.
    - BIOS 실행을 준비하는 단계이다.
2. **BIOS 실행**
    - 하드웨어 점검을 수행한다.
    - CPU, 메모리, 보조기억장치를 초기화한다.
    - 운영체제를 로딩하기 전 시스템이 정상적으로 동작 가능한지 확인한다.
3. **MBR 확인**
    - MBR은 `Master Boot Record`의 약자이다.
    - 운영체제의 위치 정보를 확인한다.
    - 어떤 저장 장치에서 운영체제를 불러올지 결정하는 데 사용된다.
4. **Boot Code 실행**
    - 부트 코드가 실행된다.
    - VBR을 호출한다.
    - 운영체제 로딩을 준비한다.
5. **운영체제 이미지 적재**
    - 운영체제 이미지를 메모리로 적재한다.
    - 파일 시스템을 표시하고 초기화한다.
6. **커널 이미지 실행**
    - 커널 이미지를 실행한다.
    - 시스템 초기화와 장치 점검을 수행한다.
7. **init 프로세스 실행**
    - init 프로세스를 실행한다.
    - 초기화 작업을 수행한다.
8. **Login Prompt 표시**
    - 로그인 프롬프트가 표시된다.
    - 사용자가 인증을 완료하면 시스템 사용이 가능해진다.
## 1.2 BIOS(Basic Input Output System)
### 1.2.1 BIOS의 역할
- 컴퓨터 부팅 초기에 실행되는 펌웨어이다.
- 하드웨어 점검을 수행한다.
- CPU, 메모리, 보조기억장치 등 주요 장치를 초기화한다.
- 입출력 처리를 담당한다.
- 운영체제 로딩 전 단계에서 필수적으로 동작한다.
- 펌웨어 형태로 저장된다.
- 비휘발성 메모리인 NVRAM 또는 ROM에 저장된다.
- 전원이 차단되어도 설정과 정보가 유지된다.
### 1.2.2 BIOS가 제공하는 기능
- 부팅 시 화면 출력
- BIOS 환경 설정 접근
- 공장 초기화 기능
- 하드웨어 점검 및 초기화
- 운영체제 로딩 준비
## 1.3 디지털 포렌식에서의 BIOS
BIOS는 운영체제보다 먼저 실행되므로 시스템의 기준 시간, 저장 매체 구성, 부팅 정보를 확인하는 단서가 된다.
### 1.3.1 BIOS가 포렌식에서 중요한 이유
| 확인 대상 | 포렌식 의미 |
| --- | --- |
| 시스템 시계 | 기준 시간 조작 여부 확인 |
| 저장 매체 구성 | HDD/SSD 장착 여부, 용량, 구성 확인 |
| 부팅 우선순위 | 어떤 장치에서 운영체제를 불러왔는지 추정 |
| 부팅 모드 | BIOS/UEFI, Legacy/Secure Boot 등 환경 확인 |
## 1.4 BIOS vs. UEFI
BIOS와 UEFI는 모두 운영체제를 구동하기 위한 필수 펌웨어이다. 다만 UEFI는 BIOS의 한계를 개선한 **차세대 펌웨어**이다.
### 1.4.1 BIOS
- 텍스트 기반 인터페이스를 제공하는 전통적인 펌웨어이다.
- 하드웨어 초기화 후 부트로더를 실행한다.
- 대용량 디스크 지원, 부팅 속도, 사용성 측면에서 한계가 있다.
### 1.4.2 UEFI
- GUI 기반 인터페이스와 대용량 디스크 지원을 제공한다.
- 부팅 과정을 간소화하고 속도를 개선한다.
- Secure Boot 등 부팅 이미지 무결성 검증 기능을 제공한다.
### 1.4.3 BIOS와 UEFI의 공통점
- 둘 다 운영체제를 구동하기 위한 펌웨어이며, 전원 인가 후 하드웨어 초기화와 부팅 준비를 담당한다.
# 2. 레지스트리 분석
## 2.1 윈도우 부팅 단계
윈도우 부팅 단계는 Boot Manager에서 시작해 Winload, 커널 초기화, 세션 관리자, 로그인 및 Explorer 초기화로 이어진다. 각 단계는 포렌식 관점에서 확인 가능한 흔적을 남길 수 있다.
### 2.1.1 Boot Manager
- BIOS 또는 UEFI가 부팅 장치를 확인한다.
- 이후 제어권을 Boot Manager에게 전달한다.
- Boot Manager는 BCD(Boot Configuration Data)를 로딩한다.
- BCD 로딩을 통해 운영체제와 부팅 모드를 결정한다.
### 2.1.2 Winload.exe / Winload.efi
- **커널, HAL, 핵심 드라이버** 등을 메모리에 로딩.
- **안전모드와 복구 모드** 등의 설정 적용.
### 2.1.3 Kernel Init
- 커널, 서브시스템을 초기화한다.
- 운영체제의 제어권이 커널로 전달된다.
### 2.1.4 Session Manager: smss.exe
- 사용자 세션을 생성한다.
- 레지스트리를 초기화한다.
- 시작 프로그램을 로딩한다.
### 2.1.5 Windows Logon / Explorer
- 사용자가 로그인을 완료한다.
- 사용자 환경이 구성된다.
- 데스크탑 인터페이스가 초기화된다.
- 이벤트 로그와 레지스트리를 통해 관련 흔적을 확인할 수 있다.
### 2.1.6 포렌식 포인트
- BCD, Winload, 이벤트 로그, 레지스트리, 세션 정보를 함께 확인하면 부팅 흐름과 사용자 로그온 흔적을 복원할 수 있다.
## 2.2 레지스트리란?
레지스트리는 윈도우 전용 데이터베이스이다. 윈도우 운영체제의 부팅, 하드웨어, 시스템 설정, 사용자 환경 정보 등을 저장한다.
### 2.2.1 레지스트리의 기본 특징
- 윈도우 전용 설정 데이터베이스이다.
- 부팅, 하드웨어, 시스템 설정, 사용자 환경 정보를 Hive 파일 단위로 저장한다.
- `Hive -> Key -> Value` 형태의 계층 구조를 가진다.
### 2.2.2 Hive 파일 단위 저장
레지스트리는 Hive 파일 단위로 저장된다. 대표적인 Hive 파일은 다음과 같다.
- `SYSTEM`
- `SOFTWARE`
- `SAM`
- `SECURITY`
- `NTUSER.DAT`
### 2.2.3 레지스트리 계층 구조
레지스트리는 다음과 같은 계층적 구조를 가진다.
```text
Hive → Key → Value
```
- **Hive**: 레지스트리의 큰 저장 단위이다.
- **Key**: 하이브 내부의 폴더 또는 경로와 같은 구조이다.
- **Value**: 실제 설정값 또는 데이터가 저장되는 단위이다.
### 2.2.4 주요 Hive / 키
| Hive / 키                   | 저장 정보                   | 포렌식 활용                  |
| -------------------------- | ----------------------- | ----------------------- |
| HKEY_LOCAL_MACHINE (HKLM)  | 시스템, 하드웨어, 드라이버, 서비스 정보 | 시스템 설정과 장치 구성 확인        |
| HKEY_CURRENT_USER (HKCU)   | 현재 로그인한 사용자의 환경 설정      | 사용자별 설정과 행위 흔적 확인       |
| HKEY_USERS (HKU)           | 사용자 계정별 프로파일 정보         | 여러 사용자 계정의 흔적 확인        |
| HKEY_CLASSES_ROOT (HKCR)   | 파일 확장자와 프로그램 연결 정보      | 특정 파일 형식과 실행 프로그램 연결 확인 |
| HKEY_CURRENT_CONFIG (HKCC) | 현재 하드웨어 프로필과 시스템 구성 정보  | 현재 부팅 환경의 하드웨어 구성 확인    |
### 2.2.5 포렌식 관점
레지스트리는 사용자의 행위와 시스템 설정을 추적할 수 있는 핵심 아티팩트이다.
| 추적 대상 | 활용 의미 |
| --- | --- |
| 시작 프로그램 / 서비스 | 악성코드 자동 실행 여부 확인 |
| 연결된 장치 | USB 등 외부 장치 사용 흔적 확인 |
| 사용자 계정 설정 | 계정별 사용 환경과 행위 추적 |
| 하드웨어 / 드라이버 | 시스템 구성 변경 흔적 확인 |
# 3. 웹 브라우저 분석
## 3.1 레지스트리 기반 사용자 행위 흔적
3차시 자료는 앞부분에서 레지스트리를 통해 확인 가능한 사용자 행위 흔적을 정리하고, 이어서 웹 브라우저 아티팩트를 다룬다.
### 3.1.1 프로그램 실행 흔적
| 아티팩트 | 경로 | 확인 내용 |
| --- | --- | --- |
| UserAssist | `HKCU\...\Explorer\UserAssist` | 프로그램 목록, 실행 횟수, 마지막 실행 시간 |
| 사용자별 자동 실행 | `HKCU\...\Run` | 현재 사용자 기준 자동 실행 항목 |
| 전체 사용자 자동 실행 | `HKLM\...\Run` | 시스템 전체 사용자 공통 자동 실행 항목 |
- 자동 실행 설정은 UserAssist, Prefetch 등과 함께 교차 검증해야 한다.
### 3.1.2 문서/파일 접근 기록
| 아티팩트 | 경로 / 저장 정보 | 확인 내용 |
| --- | --- | --- |
| RecentDocs | `HKCU\...\Explorer\RecentDocs` | 최근 열었던 문서와 파일 목록 |
| OpenSavePidlMRU / LastVisitedPidlMRU | 문서 열기/저장 시 사용한 실제 경로 정보 | 어떤 문서를 어떤 경로에서 접근했는지 추적 |
### 3.1.3 폴더 접근 기록
| 아티팩트       | 경로                                                                   | 확인 내용                                     |
| ---------- | -------------------------------------------------------------------- | ----------------------------------------- |
| BagMRU     | `HKCU\Software\Microsoft\Windows\Shell\BagMRU`                       | 폴더 열람 이력, 폴더 뷰 설정, 삭제 폴더나 USB 내부 폴더 접근 흔적 |
| TypedPaths | `HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPaths` | 탐색기 주소창에 사용자가 직접 입력한 경로                   |
### 3.1.4 USB 연결 기록
| 아티팩트 | 경로 | 확인 내용 | 결합 분석 |
| --- | --- | --- | --- |
| USBSTOR | `HKLM\SYSTEM\CurrentControlSet\Enum\USBSTOR` | USB 장치명, 제조사, 시리얼 번호, 과거 연결 장치 | 외부 저장장치 사용 여부 확인 |
| MountedDevices | `HKLM\SYSTEM\MountedDevices` | 장치가 어떤 드라이브 문자로 연결되었는지 확인 | USBSTOR와 결합해 특정 USB의 마운트 경로 추적 |
- USB 내부 폴더 접근 기록과 결합하면 파일 유출이나 외부 저장장치 사용 정황을 분석할 수 있다.
## 3.2 웹 브라우저 아티팩트
웹 브라우저 아티팩트는 사용자의 웹 사용 행위를 복원하는 데 중요한 자료이다. 방문 기록, 캐시, 다운로드 기록, 쿠키, 로그인 및 자동완성 정보가 주요 분석 대상이다.
### 3.2.1 방문 기록
| 항목 | 내용 |
| --- | --- |
| 저장 형식 | Chrome, Edge 등은 방문 기록을 SQLite DB 형태로 저장 |
| 확인 내용 | 방문 사이트, URL, 검색어 기록 |
| 시간 정보 | 마이크로초 단위 시간 기록 제공 |
| 포렌식 활용 | 정확한 타임라인 복원 |
### 3.2.2 캐시
| 항목 | 내용 |
| --- | --- |
| 저장 대상 | HTML, 이미지, JavaScript, 폰트 등 웹 리소스 |
| 확인 내용 | 캐시 시간 기록을 통한 방문 사실 |
| 포렌식 활용 | 히스토리 삭제 여부와 증거 인멸 정황 판단 |
### 3.2.3 다운로드 기록
| 항목 | 내용 |
| --- | --- |
| 저장 위치 | 웹 브라우저 DB |
| 확인 내용 | 다운로드 파일명, 저장 위치, 다운로드 시간 |
| 포렌식 활용 | 사용자가 어떤 파일을 내려받았는지 분석 |
### 3.2.4 쿠키
| 항목 | 내용 |
| --- | --- |
| 저장 정보 | 세션 유지 정보, 자동 로그인 관련 정보 |
| 확인 내용 | 브라우징 패턴, 사용자 행위 |
| 포렌식 활용 | 세션 타임값을 통한 접속 시점 추정 |
### 3.2.5 로그인 / 자동완성 정보
| 항목 | 내용 |
| --- | --- |
| 저장 정보 | 웹사이트 로그인 정보, 자동완성 데이터 |
| 포함 가능 데이터 | 이름, 주소, 이메일 등 |
| 포렌식 활용 | 계정 사용 흔적과 접속 행위 분석 |
## 3.3 교차 검증의 중요성
3차시의 핵심은 레지스트리와 웹 브라우저 DB를 함께 비교해 사용자의 행위를 복원하는 것이다.
### 3.3.1 단일 아티팩트의 한계
- 단일 아티팩트만으로 행위를 단정하면 위험하다.
- 자동 실행 레지스트리 값은 설정 여부를 보여줄 뿐 실제 실행을 직접 증명하지는 않는다.
- UserAssist, Prefetch, 브라우저 DB, 캐시, 다운로드 기록, USB 연결 기록 등을 함께 비교해야 한다.
