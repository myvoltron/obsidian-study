# 1. 문자의 저장 및 처리
## 1.1 BASE64 Encoding
### 1.1.1 Base64의 개념
- 바이너리 데이터를 문자 데이터로 안전하게 변환하는 인코딩 방식이다.
- URL, HTTP 요청, 검색 쿼리처럼 특수문자나 공백이 문제가 될 수 있는 환경에서 사용된다.

### 1.1.2 인코딩과 디코딩
| 구분 | 내용 |
| --- | --- |
| 인코딩 | 원본 바이너리 데이터를 Base64 문자 집합으로 변환 |
| 디코딩 | Base64 문자열을 원본 바이너리 데이터로 복원 |
| 핵심 | 전송 중 깨지기 쉬운 데이터를 안전한 문자 형태로 바꿈 |

## 1.2 Character Code
### 1.2.1 문자 코드의 역할
- 문자를 컴퓨터가 저장할 수 있는 숫자와 비트 패턴으로 매핑하는 표준이다.
- 문자 코드가 다르면 같은 바이트열도 다른 문자로 해석될 수 있다.

### 1.2.2 주요 문자 코드
| 구분 | 특징 |
| --- | --- |
| ASCII | 영문 중심 7비트 문자 체계, 128개 기호 표현 |
| Extended ASCII | 8비트로 확장하여 256개 기호 표현 |
| Unicode | 전 세계 문자를 통합적으로 표현하기 위한 표준 |
| UTF-8 | 1~4바이트 가변 길이 방식, 현대 시스템에서 널리 사용 |
| UTF-32 | 4바이트 고정 길이 방식 |

## 1.3 한글 문자 코드
### 1.3.1 한글 인코딩 방식
| 방식 | 특징 | 한계 / 활용 |
| --- | --- | --- |
| 조합형 | 초성·중성·종성을 조합해 글자 생성 | 모든 조합 표현은 가능하지만 모양 품질이 낮음 |
| 완성형 | 자주 쓰는 글자를 완성된 형태로 저장 | 정의되지 않은 글자는 표현 어려움 |
| 확장 완성형 / EUC-KR | 완성형의 부족한 글자를 보완한 2바이트 계열 | 국내 환경에서 많이 사용 |
| Unicode / UTF-8 | 다양한 언어와 한글을 함께 표현 | 현대 시스템의 기본 선택지 |

# 2. 시간 정보 표현
## 2.1 시간 정보의 의미
### 2.1.1 포렌식에서의 시간
- 생성, 수정, 접근, 실행 시각은 사건 타임라인 복원의 핵심 단서이다.
- 시스템 시간이 실제 시간과 다를 수 있으므로 PC 설정 시각과 KST/UTC 기준을 함께 확인해야 한다.

### 2.1.2 시간 기준
| 기준 | 내용 |
| --- | --- |
| GMT | 경도 0도를 기준으로 한 평균 태양시 |
| UTC | 1972년부터 쓰인 국제 표준시, 원자 시계 기반 |
| KST | UTC+9, 한국 표준시 |

## 2.2 파일 시스템의 시간 저장 방식
### 2.2.1 주요 포맷
| 파일 시스템 | 시간 저장 방식 | 특징 |
| --- | --- | --- |
| FAT | MS-DOS Date/Time | 4바이트 구조, 2초 단위 정밀도 |
| NTFS | FILETIME | 8바이트 구조, 100나노초 단위 |

### 2.2.2 MS-DOS Date/Time
| 필드 | 비트 수 | 해석 |
| --- | --- | --- |
| 연도 | 7bit | 1980 + 값 |
| 월 | 4bit | 1~12 |
| 일 | 5bit | 1~31 |
| 시 | 5bit | 0~23 |
| 분 | 6bit | 0~59 |
| 초 | 5bit | 2초 단위 |

### 2.2.3 Endian과 시간 해석
- Intel 계열 시스템은 Little-Endian을 사용하므로 바이트 순서를 뒤집어 해석해야 하는 경우가 많다.
- 예를 들어 `A472723B`가 MS-DOS Date/Time 값이라면 저장 순서와 포맷 규칙을 먼저 확인한 뒤 날짜와 시간 필드로 분리한다.

### 2.2.4 FILETIME
- NTFS에서 사용하는 시간 포맷이다.
- 1601년 1월 1일 00:00:00 UTC를 기준으로 100나노초 단위를 누적해 저장한다.
- MS-DOS Date/Time보다 표현 범위와 정밀도가 높다.

# 3. 파일 시그니처 분석
## 3.1 파일 시그니처의 의미
### 3.1.1 개념
- 파일 내부에 기록된 고유 패턴으로 파일의 실제 종류를 식별하는 단서이다.
- 확장자는 쉽게 바꿀 수 있으므로 파일 종류 판단은 내부 구조와 시그니처를 기준으로 해야 한다.

### 3.1.2 파일 포맷 구조
| 영역 | 내용 |
| --- | --- |
| Header | 파일 종류 식별용 시그니처, 크기, 메타데이터 위치 등 |
| Metadata | 파일 설명 정보, 생성 도구, 시간 정보 등 |
| Body | 실제 데이터 내용 |

### 3.1.3 포렌식 활용
- 확장자를 위장한 파일을 식별한다.
- 파일 손상 또는 삭제 후 복구 가능성을 판단한다.
- 파일 카빙 시 시작·종료 패턴을 찾는 기준으로 사용한다.
