# 1. 문자의 저장 및 처리
## 1.1 BASE64 Encoding
### 1.1.1 Base64의 개념
- 바이너리 데이터를 문자 데이터로 안전하게 변환하는 인코딩 방식이다. 이름이 의미하듯 64개의 문자를 사용하여 데이터를 표현한다.
### 1.1.2 인코딩 원리
- 원본 데이터를 8비트 단위로 쪼갠 후, 각 8비트 블록을 Base64 문자 1개와 매핑하는 방식으로 진행된다. 이는 숫자 2, 4, 8, 16, 32, 64 진법과 유사한 확장 원리를 가진다.
### 1.1.3 디코딩 원리
- 인코딩의 역과정으로 Base64 문자를 8비트 바이너리 데이터로 변환하여 원본 데이터와 동일하게 복원할 수 있다. 핵심은 64개의 문자를 사용하며, 1개의 문자가 8비트 데이터에 매핑된다는 점이다.
### 1.1.4 활용 예시
- URL, HTTP 요청 메시지, 검색 쿼리(Query) 등에 널리 쓰인다. **공백이나 특수문자**를 안전하게 처리하여 전송 오류를 방지하고, 사용자 입력 문자를 시스템에 안전하게 인코딩하여 전달하는 역할을 한다.
## 1.2 Character Code (문자 코드)
### 1.2.1 문자 코드의 역할
- 사용자가 입력한 문자를 컴퓨터가 이해할 수 있는 바이너리 패턴으로 변환하는 과정이다. 이를 위해 ASCII, Unicode 등과 같은 표준화된 매핑 테이블이 필요하다. 이러한 표준화의 주요 목적은 컴퓨터 간 문자의 일관성을 유지하고 다양한 언어를 지원하기 위함이다.
### 1.2.2 ASC(American Standard Code)
- 미국 표준협회에서 지정한 표준 셋으로, 알파벳, 특수문자, 제어문자, 숫자 등을 동일한 방법으로 바이너리 데이터에 매핑시킨 테이블이다. 이를 표준으로 지정함으로써 기기 간 데이터 전송이나 복사 및 로딩 시 동일한 문자로 보이게 하는 역할을 한다.
	- 제어부호 33자, 그래픽 기호 33자, 숫자 10자, 알파벳 대소문자 52자
	- 통신의 시작과 종료, 라인 피드 등을 표시
	- 7비트로 구성 -> 128개 기호
### 1.2.3 ASCII
- ASC 지정 후 이를 확장한 체계이다. 초기 문자셋은 영문 중심의 7비트(128개 문자)로 구성되어 알파벳, 숫자, 특수문자, 제어문자를 포함했다. 이후 확장 ASCII(ASCII2)의 도입으로 8비트($2^8$, 256종 기호) 전체를 문자로 표현할 수 있게 되었으며, 문서 작성 및 통신 프로토콜에 주로 활용된다.
## 1.3 Unicode (유니코드)
### 1.3.1 개념
- **전 세계의 모든 문자**를 표현(Universal Character Set, UCS)하기 위한 표준 규격이다. 원래는 32비트지만 대부분 **16비트** 기반의 BMP(Basic Multilingual Plane) 영역을 사용하며, 각 문자에 고유한 16진수 번호를 부여하는 '유니코드 포인트'를 활용한다.
### 1.3.2 인코딩 방식
- **UCS-2**: 2바이트(16비트) 고정 길이로, BMP 영역의 문자를 표현한다.
- **UCS-4**: 4바이트(32비트) 고정 길이로, 모든 유니코드 문자를 표현한다.
- **UTF-8**: 1~4바이트의 **가변 길이 인코딩 방식**을 사용하며, 효율성과 호환성이 매우 우수하여 현대 컴퓨터 시스템의 표준으로 널리 사용된다.
- **UTF-32**: UCS-4와 동일한 개념을 가지는 고정 길이 표현 방식이다.
# 2. 문자의 저장 및 처리 및 시간 정보 표현
## 2.1 Character Code: 한글 문자 코드
### 2.1.1 한글 문자의 특징
- 초성, 중성, 종성으로 구성되는 과학적 구조를 가진다. 이들의 조합 방식에 따라 글자의 모양이 변하므로 경우의 수가 매우 많다. 따라서 알파벳 대비 폰트 제작과 인코딩 과정이 복잡하다.
### 2.1.2 한글 인코딩 방식
- **조합형 코드**: 타자기를 기반으로 한 초기 버전으로, 초·중·종성 조합으로 글자를 생성한다. 이론상 모든 한글 조합을 표현할 수 있지만 글자 모양이 미려하지 않다는 단점이 있다.
- **완성형 코드**: 자주 사용하는 글자만 미리 완성된 형태로 표준화하여 미려한 표현이 가능하지만, 사전에 정의되지 않은 일부 글자는 조합이 불가능하다는 한계가 있어 확장이 필요했다.
- **확장 완성형**: 기존 완성형 코드로 표현 불가능했던 한글을 추가로 포함시켰다. 유니코드와 같이 넓은 캐릭터셋을 활용할 수 있으며, 2바이트(16비트)를 기반으로 하는 **EUC-KR**이 그 표준이다.
    - 0xA1 ~ 0xFE 값을 가짐
## 2.2 시간 정보 표현 방식
### 2.2.1 시간 정보의 중요성
- 디지털 포렌식에서 시간(Time)은 매우 중요한 데이터 종류이다. 일반적으로 연, 월, 일, 시, 분, 초로 표현되며, 이는 디지털 데이터인 0101 패턴 형태로 인코딩되어 저장되어야 한다.
### 2.2.2 시간 표현 기준
- **GMT (Greenwich Mean Time)**: 경도 0을 기준으로 하는 평균 태양시로 1972년까지 세계 표준시로 사용되었다.
- **UTC (Coordinated Universal Time)**: 1972년 1월 1일부터 표준시로 도입되었다. 세슘 원자 진동수를 측정하는 원자 시계를 기준으로 하며, 현재 국제 표준시로 사용되고 있다. -> 서울은 UTC + 9
### 2.2.3 절대 시간(Absolute Time)의 필요성
- 기기 내부의 시스템 시각은 실제 시각과 다르게 설정되어 있을 수 있다. 따라서 PC 설정 시각과 KST(한국표준시)를 함께 기록하여 절대 시간을 추정해야 하며, 이는 증거 타임라인 복원에 필수적이다.
## 2.3 디지털 포렌식과 시간정보
### 2.3.1 컴퓨터 시스템 시간
- 컴퓨터는 내부 시계를 기반으로 시간을 기록하며, 파일의 생성, 수정, 접근 시점을 기록한다. 파일 시스템마다 시간을 기록하는 방식이 각기 다르다.
### 2.3.2 파일 시스템의 시간 인코딩 방식
- 생성·수정·접근 시각을 바이너리로 저장하며 포맷 종류별로 저장 방식에 차이가 있다.
    - **FAT**: 4바이트 공간을 사용하며 MS-DOS Time 방식으로 저장한다.
    - **NTFS**: 8바이트 공간을 사용하며 FILE TIME 방식을 사용한다.
    - 저장 공간이 클수록 더 정확하고 정밀한 시간 표현이 가능하다.
# 3. 시간 정보 표현 및 파일 시그니처
## 3.1 컴퓨터의 시간 저장 형식
### 3.1.1 MS-DOS Date/Time
- 오래된 운영체제인 MS-DOS에서 사용되던 시간 저장 방식이다. 총 2바이트(16비트) 공간에 각각 날짜와 시간을 모두 압축하여 저장한다. 표현 가능한 범위는 **1980년부터 2107년 12월 31일 23시 59분 58초** 까지이며, 저장 공간의 한계로 인해 정확도가 2초 단위로 낮다.
### 3.1.2 비트 구성 (16비트 분할)
- 연도(Y): 7비트 ($1980 + \text{값}$)
- 월(M): 4비트 (1~12월)
- 일(D): 5비트 (1~31일)
- 시(H): 5비트 (0~23시)
- 분(M): 6비트 (0~59분)
- 초(S): 5비트 (2초 단위로 기록됨) 어쩔 수 없는 타협,,,,
- 문제: `A472723B`를 해석해보자! 이는 MS-DOS Date/Time 형식으로 저장되어 있다. 
### 3.1.3 Endian의 중요성
- CPU에 따라 저장 및 해석 방식이 달라진다. 인텔(Intel) CPU는 Little Endian 방식을 사용하므로 `A472723B`라는 데이터는 바이트 역순인 `3B 72 72 A4`로 재배치한 후 해석해야 올바른 값을 얻을 수 있다.
- -> `0011 1011 0111 0010 0111 0010 1010 0100`: 2009년 11월 18일 ... 8초
### 3.1.4 MS-DOS Date/Time의 한계
- 16비트라는 제한된 공간은 표현 범위와 **정확도에 큰 제약**(2초 단위, 2107년까지만 표현 가능)을 가져온다. 시간 정밀도가 낮아 정교한 포렌식 분석에 사용하기에는 뚜렷한 한계가 존재한다.
### 3.1.5 FILETIME (차세대 시간 포맷)
- MS-DOS Date/Time의 차세대 버전으로 NTFS 파일 시스템에서 사용된다. **8바이트**(64비트) 구조를 지녀 훨씬 정밀한 시간 기록이 가능하다. 기준 시점은 1601년 1월 1일 자정(UTC)이며 단위는 100나노초($10^{-7}$초)이다. 표현 범위와 정확도가 대폭 향상되어 파일의 생성, 수정, 접근 시각을 매우 정밀하게 추적할 수 있다.
## 3.2 파일 시그니처 (File Signature) 분석
### 3.2.1 파일 시그니처란?
- 각 파일만이 고유하게 갖는 특징이다. 파일 내부에 기록된 고유 패턴을 통해 해당 파일의 실제 종류를 판별한다. 확장자와는 무관하여, 확장자를 `.txt`에서 `.hwp`로 변경해도 아이콘만 변할 뿐 파일의 실제 정체는 동일하게 유지된다.
### 3.2.2 파일 시그니처 분석이 필요한 이유
- 확장자는 단순히 운영체제가 어떤 프로그램을 실행할지 결정하는 용도에 불과하다. 악의적인 사용자는 확장자 변경을 통해 파일을 쉽게 위장할 수 있다. 따라서 실제 포렌식 조사에서는 파일 내부 구조(시그니처)를 기준으로 파일의 종류를 판별하는 것이 필수적이다.
### 3.2.3 파일 포맷과 시그니처
- 대부분의 파일은 **일정한 파일 포맷 구조**를 가진다.
    - **Header (헤더)**: 파일의 시작 부분으로, 파일 종류 식별용 시그니처가 포함되며 파일 크기나 메타데이터 위치 정보 등이 저장된다.
    - **Metadata (메타데이터)**: 파일을 설명하는 부가 정보 영역으로, 파일 포맷에 따라 존재할 수도 있고 없을 수도 있다.
    - **Body (데이터 영역)**: 파일의 내용에 해당하는 실제 데이터가 저장되는 구역이다.
