# 디지털포렌식 시험 암기 포인트

시험 유형을 모를 때는 **정의, 비교, 절차, 구조, 포렌식 의미**를 우선 암기한다.

# 1. 최우선 암기

## 1.1 디지털 포렌식 5대 원칙
| 원칙 | 암기 포인트 |
| --- | --- |
| 정당성 | 적법한 절차로 증거를 수집해야 함 |
| 원본성 | 원본과 동일한 상태로 재현 가능해야 함 |
| 무결성 | 수집 이후 변조되지 않았음을 입증해야 함 |
| 신뢰성 | 검증된 도구와 전문성을 바탕으로 분석해야 함 |
| 연계 보관성 | 획득부터 제출까지 증거 취급 이력을 관리해야 함 |

암기문: **정원무신연**

## 1.2 디지털 포렌식 조사 7단계
1. 사전 준비
2. 증거 수집
3. 데이터 보존
4. 조사 분석
5. 정밀검토 및 보고
6. 법적 대응 및 증언
7. 사후 대응

암기문: **사수보분보고법사**

## 1.3 디지털 증거의 증거능력 요건
| 요건 | 핵심 |
| --- | --- |
| 원본성 | 법정 제출 자료가 원 매체 데이터와 동일해야 함 |
| 무결성 | 수집, 이송, 분석, 제출 과정에서 변경이 없음을 입증 |
| 신뢰성 | 절차, 도구, 인력, 시설이 믿을 만해야 함 |

자주 같이 나오는 것: **해시값**, **전자서명**, **Chain of Custody**, **검증된 도구**.

## 1.4 증거능력 vs 증명력
| 구분 | 의미 |
| --- | --- |
| 증거능력 | 법정에서 증거로 사용할 수 있는 법률상 자격 |
| 증명력 | 그 증거가 실제로 얼마나 믿을 만한지의 정도 |

판단 흐름: **증거능력 판단 → 증명력 평가**

## 1.5 디지털 증거의 특성
| 특성 | 암기 포인트 |
| --- | --- |
| 매체 독립성 | 정보 자체가 본질이므로 원본과 사본 구별이 어려움 |
| 비가시성/비가독성 | 도구로 해석해야 의미 파악 가능 |
| 변조/복제 용이성 | 해시값으로 원본성·무결성 입증 필요 |
| 대규모성/초국경성 | 증거가 여러 시스템·국가에 분산될 수 있음 |

# 2. 증거 수집과 이미징

## 2.1 디스크 이미징 vs 라이브 포렌식
| 구분 | 디스크 이미징 | 라이브 포렌식 |
| --- | --- | --- |
| 대상 | 비휘발성 저장매체 | 휘발성 데이터 |
| 예시 | HDD, SSD, 파일시스템 | RAM, 프로세스, 네트워크 연결 |
| 안정성 | 높음 | 낮음 |
| 긴급성 | 낮음 | 높음 |
| 손상 위험 | 낮음 | 높음 |

## 2.2 디스크 이미징
- 저장매체 전체를 **비트 단위**로 복제하는 과정이다.
- **Bit Stream Image**라고도 한다.
- 원본 데이터뿐 아니라 **메타데이터, 삭제 데이터, 빈 공간 데이터**까지 포함한다.
- 도구: `FTK Imager`, `EnCase`, `dd`, `dcfldd`
- 핵심 장치: **Write-Blocker**, **해시값**

## 2.3 라이브 포렌식
- 구동 중인 시스템에서 데이터를 수집하고 분석한다.
- 전원 차단 시 사라지는 **휘발성 데이터**를 우선 수집한다.
- 수집 대상: 메모리, 프로세스, 네트워크 연결, 사용자 세션, CPU/RAM 사용 현황
- 도구: `DumpIt`, `Volatility`, `Rekall`, `Sysinternals Suite`, `Wireshark`
- 원칙: **사라지기 쉬운 데이터부터 수집**, **시스템 변경 최소화**, **도구/명령/순서 기록**

## 2.4 물리적 획득 vs 논리적 획득
| 구분 | 물리적 획득 | 논리적 획득 |
| --- | --- | --- |
| 관점 | 저장장치 전체 | 운영체제·파일시스템 |
| 단위 | 비트/저수준 저장 영역 | 사용자 접근 가능 파일 |
| 장점 | 삭제·숨김 데이터 확보 가능 | 빠르고 손상 가능성 낮음 |
| 한계 | 시간과 난이도 높음 | 삭제 데이터·숨김 영역 누락 가능 |

관련 기법:
- **CHIP-OFF**: 저장 칩을 분리해 추출하는 침습 방식
- **JTAG**: 디버깅 포트를 이용해 저수준 메모리에 접근하는 비침습 방식

# 3. 데이터 표현과 해석

## 3.1 진법과 단위
| 항목 | 암기 |
| --- | --- |
| bit | 0 또는 1 하나 |
| Byte | 8bit |
| 16진수 1자리 | 4bit |
| 1Byte 표현 범위 | 0~255 |

포렌식에서 16진수를 많이 쓰는 이유: **Raw Data와 1:1 매핑이 쉽기 때문**.

## 3.2 Big-Endian vs Little-Endian
| 구분 | 저장 방식 | 특징 |
| --- | --- | --- |
| Big-Endian | 큰 자릿값 바이트 먼저 | 사람이 읽는 순서와 유사, 네트워크에서 자주 사용 |
| Little-Endian | 작은 자릿값 바이트 먼저 | Intel/AMD 계열 해석 시 중요 |

## 3.3 음수와 소수 표현
| 주제 | 암기 포인트 |
| --- | --- |
| 부호-크기 | 최상위 비트가 부호, +0/-0 문제 |
| 1의 보수 | 비트 반전, 0이 두 개 |
| 2의 보수 | 비트 반전 후 1, 현대 컴퓨터 표준 |
| 부동 소수점 | 부호, 지수, 가수 구조 |

## 3.4 문자와 인코딩
| 구분 | 핵심 |
| --- | --- |
| Base64 | 바이너리 데이터를 안전한 문자 형태로 변환 |
| ASCII | 7비트, 128개 기호 |
| Extended ASCII | 8비트, 256개 기호 |
| Unicode | 전 세계 문자 통합 표준 |
| UTF-8 | 1~4바이트 가변 길이 |
| UTF-32 | 4바이트 고정 길이 |

## 3.5 시간 정보
| 파일시스템 | 시간 방식 | 핵심 |
| --- | --- | --- |
| FAT | MS-DOS Date/Time | 4바이트, 2초 단위 |
| NTFS | FILETIME | 1601-01-01 UTC 기준, 100나노초 단위 |

주의: 포렌식 타임라인은 **시스템 시간 조작**, **UTC/KST 차이**, **Endian**을 함께 확인한다.

## 3.6 파일 시그니처
- 파일 내부의 고유 패턴으로 실제 파일 종류를 식별한다.
- 확장자는 쉽게 바꿀 수 있으므로 **Header, Metadata, Body** 구조와 시그니처를 확인한다.
- 활용: 확장자 위장 탐지, 손상/삭제 파일 복구 판단, 파일 카빙.

# 4. 스테가노그래피

## 4.1 정의와 구성
| 용어 | 의미 |
| --- | --- |
| 스테가노그래피 | 데이터가 존재한다는 사실 자체를 숨기는 정보 은닉 기술 |
| Carrier | 데이터를 숨길 매개체 |
| Payload | Carrier 안에 실제로 숨기는 비밀 데이터 |

## 4.2 크립토그래피 vs 스테가노그래피
| 구분 | 핵심 |
| --- | --- |
| 크립토그래피 | 내용은 숨기지만 암호화된 데이터의 존재는 보임 |
| 스테가노그래피 | 데이터 존재 자체를 숨김 |

## 4.3 LSB 기법
- RGB 각 색상은 0~255, 즉 8bit 값을 가진다.
- 한 픽셀은 보통 `R8 + G8 + B8 = 24bit`이다.
- **최하위 비트(LSB)**를 조작해 비밀 데이터를 삽입한다.
- 사람 눈은 1~2 정도의 색상 값 변화를 거의 구별하지 못한다.
- 한계: 순차 삽입은 통계 분석으로 탐지될 수 있고, 과도 삽입 시 육안으로도 티가 날 수 있다.

# 5. 윈도우 아티팩트

## 5.1 부팅 흐름
기본 흐름:
```text
전원 인가 → BIOS/UEFI → MBR/Boot Code → OS 이미지 적재 → 커널 실행 → init/session → Login
```

윈도우 세부 흐름:
```text
Boot Manager → Winload.exe/efi → Kernel Init → smss.exe → Windows Logon → Explorer
```

## 5.2 BIOS/UEFI 포렌식 의미
| 확인 대상 | 의미 |
| --- | --- |
| 시스템 시계 | 기준 시간 조작 여부 |
| 저장 매체 구성 | HDD/SSD 장착 여부와 용량 |
| 부팅 우선순위 | 어느 장치로 부팅했는지 |
| 부팅 모드 | BIOS/UEFI, Legacy/Secure Boot |

## 5.3 레지스트리
- 윈도우 전용 설정 데이터베이스이다.
- 구조: **Hive → Key → Value**
- 주요 Hive: `SYSTEM`, `SOFTWARE`, `SAM`, `SECURITY`, `NTUSER.DAT`

| Hive/키 | 포렌식 활용 |
| --- | --- |
| HKLM | 시스템, 하드웨어, 드라이버, 서비스 정보 |
| HKCU | 현재 로그인 사용자 환경 설정 |
| HKU | 사용자 계정별 프로파일 |
| HKCR | 파일 확장자와 프로그램 연결 |
| HKCC | 현재 하드웨어 프로필 |

## 5.4 사용자 행위 아티팩트
| 아티팩트 | 확인 내용 |
| --- | --- |
| UserAssist | 프로그램 목록, 실행 횟수, 마지막 실행 시간 |
| Run | 자동 실행 항목 |
| RecentDocs | 최근 열었던 문서와 파일 |
| OpenSavePidlMRU / LastVisitedPidlMRU | 문서 열기/저장 경로 |
| BagMRU | 폴더 열람 이력, 삭제 폴더/USB 내부 폴더 접근 흔적 |
| TypedPaths | 탐색기 주소창에 직접 입력한 경로 |
| USBSTOR | USB 장치명, 제조사, 시리얼 번호 |
| MountedDevices | 장치와 드라이브 문자 매핑 |

## 5.5 웹 브라우저 아티팩트
| 아티팩트 | 확인 내용 |
| --- | --- |
| 방문 기록 | URL, 검색어, 방문 시간 |
| 캐시 | 웹 리소스, 히스토리 삭제 여부 판단 |
| 다운로드 기록 | 파일명, 저장 위치, 다운로드 시간 |
| 쿠키 | 세션 유지, 접속 시점 추정 |
| 로그인/자동완성 | 계정 사용 흔적 |

핵심: 단일 아티팩트만으로 단정하지 말고 **UserAssist, Prefetch, 브라우저 DB, 캐시, 다운로드 기록, USB 기록**을 교차 검증한다.

# 6. 시스템 로그와 안티 포렌식

## 6.1 로그
| 구분 | 핵심 |
| --- | --- |
| 시스템 로그 | 시스템과 사용자 행위를 시간 순으로 기록 |
| 윈도우 보안 로그 | 로그인 성공/실패, 파일 접근, 보안 이벤트 |
| 윈도우 시스템 로그 | 부팅, 장치 연결, 오류 |
| 응용 프로그램 로그 | 프로그램 오류, 실행 상태 |
| 웹 로그 | 방문 기록, 캐시, 쿠키 |

리눅스/유닉스 로그:
- `acct/pact`: 명령 실행 기록
- `lastlog`: 최근 로그인
- `loginlog`: 로그인 실패
- `messages`: 시스템 메시지, 부팅 메시지
- `sulog`: `su` 명령 사용 기록

## 6.2 안티 포렌식
- 포렌식 분석을 방해하거나 무력화하는 기술이다.
- 목적: **증거 삭제, 증거 은닉, 증거 훼손**

| 기법 | 내용 |
| --- | --- |
| 데이터 영구 삭제 | Disk Wiping, Degausser |
| 자동 삭제 프로그램 | 특정 조건·주기에 따라 덮어쓰기/삭제 |
| 암호화 | 키 없이는 내용 확인 어려움 |
| 스테가노그래피 | 데이터 존재 자체 은닉 |

# 7. 저장매체와 스토리지

## 7.1 휘발성 vs 비휘발성
| 구분 | 예시 | 포렌식 의미 |
| --- | --- | --- |
| 휘발성 | RAM | 전원 차단 전 우선 수집 |
| 비휘발성 | HDD, SSD | 이미징과 파일시스템 분석 대상 |

## 7.2 SSD 포렌식 주의점
| 기능 | 의미 | 포렌식 영향 |
| --- | --- | --- |
| Wear Leveling | 기록 위치를 분산해 셀 수명 연장 | 논리 주소와 물리 위치가 단순 대응되지 않을 수 있음 |
| TRIM | 삭제 블록을 SSD 컨트롤러에 알림 | 삭제 데이터 복구 가능성이 낮아질 수 있음 |

## 7.3 RAID
| 레벨 | 방식 | 장점 | 한계 |
| --- | --- | --- | --- |
| RAID 0 | 스트라이핑 | 속도 향상 | 디스크 하나만 고장 나도 전체 손실 |
| RAID 1 | 미러링 | 복구 안정성 | 사용 가능 용량 감소 |
| RAID 5 | 스트라이핑 + 패리티 | 속도와 안정성 균형 | 최소 3개 디스크 필요 |
| RAID 1+0 | 미러링 + 스트라이핑 | 속도와 안정성 모두 확보 | 비용 증가 |

## 7.4 숨겨진 영역
| 구분 | 의미 |
| --- | --- |
| HPA | Host Protected Area, 디스크 끝부분 예약 숨김 영역 |
| DCO | Device Configuration Overlay, 제조사 수준에서 용량/기능 제한 |

이미징 전 HPA/DCO를 확인하지 않으면 증거 영역이 누락될 수 있다.

# 8. 파일시스템

## 8.1 기본 구조
```text
디스크 → 파티션 → 파일시스템 → 파일 구조
```

| 개념 | 의미 |
| --- | --- |
| 섹터 | 저장장치의 물리적 최소 단위, 보통 512B |
| 클러스터 | 파일시스템의 논리적 할당 단위 |
| 메타데이터 | 생성/수정/접근 시간, 크기 등 파일 속성 |
| Data Area | 실제 파일 내용 |
| Meta Area | 파일 속성 및 관리 정보 |

## 8.2 MBR vs GPT
| 구분 | MBR | GPT |
| --- | --- | --- |
| 위치/구조 | 디스크 앞 512B | UEFI 기반 최신 파티션 |
| 용량 | 최대 2TB | 최대 9.4ZB |
| 파티션 수 | 최대 4개 | 사실상 제한 없음 |
| 안정성 | 앞부분 한 곳에 정보 저장 | 앞/뒤에 정보 저장, CRC 지원 |
| 시그니처 | `0x55AA` | GUID 기반 |

MBR 구조:
```text
Boot Code 445B + Partition Table Entry 64B + Signature 2B
```

## 8.3 슬랙 공간
| 구분 | 의미 |
| --- | --- |
| RAM Slack | 마지막 섹터에서 실제 파일 데이터 뒤부터 섹터 끝까지 |
| File Slack | 마지막 섹터 이후부터 할당된 클러스터 끝까지 |

포렌식 의미: 이전 파일 데이터나 삭제 데이터 조각이 남을 수 있다.

## 8.4 FAT 파일시스템
전체 구조:
```text
Boot Sector → Reserved Sector → FAT1/FAT2 → Root Directory → Data Area
```

| 항목 | 핵심 |
| --- | --- |
| FAT | File Allocation Table, 클러스터 상태와 다음 위치 기록 |
| FAT1 | 메인 FAT |
| FAT2 | 백업 FAT |
| Directory Entry | 파일명, 확장자, 속성, 시간, 시작 클러스터, 파일 크기 |
| Directory Entry 크기 | 32바이트 |
| Short File Name | 8.3 규칙 |
| LFN | Long File Name, 유니코드, 여러 Directory Entry 사용 |

FAT Entry 값:
```text
0x00000000 → 비어 있는 클러스터
다음 클러스터 번호 → 연결 정보
0x0FFFFFFF → EOF
```

삭제 흔적:
- Directory Entry의 파일 이름 첫 글자: `0xE5`
- 새 엔트리 시작: `0x00`

## 8.5 NTFS 파일시스템
| 항목 | 핵심 |
| --- | --- |
| VBR | Volume Boot Record, BPB와 부트 코드 포함 |
| BPB | BIOS Parameter Block, 파일시스템 레이아웃 정보 |
| MFT | Master File Table, 파일·디렉토리 메타데이터 핵심 |
| `$BITMAP` | 사용/미사용 엔트리 또는 클러스터 관리 |
| `$DATA` | 실제 데이터 속성 |
| `$LogFile` | 변경 사항 기록 |
| `$UsnJrnl` | 변경 저널 |

NTFS 특징:
- FAT의 테이블 역할을 **MFT**가 수행한다.
- 하나의 파일은 하나의 MFT 엔트리로 관리된다.
- 주요 속성: `STANDARD_INFORMATION`, `FILE_NAME`, `DATA`
- 안정성, 보안, 백업 중심 설계.
- ADS, EFS, BitLocker, VSS를 지원한다.

접근 흐름:
```text
VBR → BPB에서 MFT 시작 위치 확인 → $MFT 접근 → $DATA로 구조 파악 → 파일/디렉토리 탐색
```

# 9. 파일 복구와 카빙

## 9.1 삭제 복구 핵심
- FAT과 NTFS 모두 기본 철학은 **데이터는 남고 인덱스만 변경된다**이다.
- FAT은 FAT Entry와 Directory Entry를 본다.
- NTFS는 MFT, `$BITMAP`, `FILE_NAME`, 파일 크기, `$DATA`를 본다.

## 9.2 파일 카빙
- 메타데이터 없이 파일을 복구하는 기법이다.
- 파일 시그니처 기반으로 복구한다.
- 시그니처는 보통 **Header와 Footer**를 의미한다.
- JPG, GIF 등 고유 패턴을 활용한다.

## 9.3 검색 기술
| 검색 | 의미 |
| --- | --- |
| 일반 검색 | 키워드, 파일명, 속성, 내부 문자열 검색 |
| 해시 검색 | 알려진 파일 해시 DB와 비교, 정상 파일 제외에 유용 |
| 슬랙 검색 | Slack Space에서 잔존 데이터 탐색 |

분석 흐름:
```text
검색 → 통계 → 시각화
```
