# 1차시: FAT 파일시스템 생성
## FAT 파일시스템 내부 구조
### FAT 파일시스템 전체 구조
#### FAT 구조
- FAT 파일시스템은 다음 순서로 구성된다.
```text
Boot Sector → Reserved Sector → FAT1/FAT2 → Root Directory → Data Area
```
- 파일시스템은 Top-Down 방식으로 접근하여 구조를 이해한다.
#### Boot Sector
- 파일시스템의 시작 영역이다.
- `BIOS Parameter Block(BPB)`을 포함한다.
- BPB에는 다음 정보가 포함된다.
    - 섹터 크기
    - 클러스터 크기
    - FAT 위치
    - Root Directory 위치
    - 파일시스템 레이아웃 정보
- `Boot Code`를 포함한다.
- Boot Code는 부팅 시 실행되는 코드이다.
#### Reserved Sectors
- Boot Sector 이후부터 FAT 시작 전까지의 **예약 공간**이다.
- 메타데이터 및 확장 구조 저장 용도로 사용된다.
#### FAT1 / FAT2
- FAT1과 FAT2는 동일한 구조를 가진다.
- FAT1은 메인 FAT이다.
- FAT2는 백업 FAT이다.
- FAT의 역할은 클러스터 간 연결 정보를 관리하는 것이다.
- 클러스터 간 연결 정보는 꼬리에 꼬리를 무는 구조이다.
- FAT2는 FAT1 손상 시 복구를 위한 백업 테이블이다.
#### FAT Entry 값
- 각 FAT 엔트리는 다음 클러스터 위치를 저장한다.
- 주요 값은 다음과 같다.
```text
0x00000000 → 비어 있는 클러스터
다음 클러스터 번호 → 연결 정보
0x0FFFFFFF → 파일의 끝, EOF
```
### Root Directory & Data Area
#### Root Directory
- 포맷 시 자동 생성되는 기본 디렉토리이다.
- 파일 및 하위 디렉토리의 Directory Entry를 저장한다.
- 위치가 고정된다.
#### Directory Entry
Directory Entry에는 다음 정보가 저장된다.
- 파일명
- 확장자
- 속성(Attribute)
- 생성 시간 정보
- 수정 시간 정보
- 시작 클러스터 번호
- 파일 크기 정보
#### Data Area
- 폴더와 파일의 실제 데이터가 저장되는 영역이다.
- 클러스터 단위로 저장된다.
- 위치는 FAT을 통해 추적 가능하다.
### Directory Entry 구조 및 LFN
#### 기본 Directory Entry
- 기본 Directory Entry의 크기는 32바이트이다.
#### Short File Name
- Short File Name은 8.3 규칙을 따른다.
- 파일 이름은 8자리이다.
- 확장자는 3자리이다.
#### Long File Name
- Long File Name은 **LFN**이라고 한다.
- 8.3 규칙의 한계를 극복하기 위한 구조이다.
- 유니코드를 지원한다.
- 유니코드는 문자당 2바이트를 사용한다.
- 여러 개의 Directory Entry를 사용하여 파일 이름을 저장한다.
#### 엔트리 구분
- `Attribute = 0x0F`이면 LFN 엔트리이다.
- `Attribute = 0x20`이면 일반 파일 엔트리이다.
## FAT 파일시스템 생성
### 파일 생성 과정 예시
#### 예시 조건
- 클러스터 크기는 4096B이다.
- `file1.dat`의 크기는 6000B이다.
#### 파일 생성 흐름
- 포맷 후 비어 있는 파일시스템에서 `dir1` 디렉토리를 생성한다.
- `dir1` 안에 파일 생성 요청이 발생한다.
- Directory Entry를 탐색한다.
- 삭제된 엔트리는 `0xE5`로 표시된다.
- 새 엔트리 시작은 `0x00`으로 표시된다.
- FAT에서 빈 FAT Entry인 `0x00`을 검색한다.
- 첫 클러스터로 190번을 확보한다.
- 6000B 파일은 클러스터 2개가 필요하다.
- 190번 클러스터에 4096B를 저장한다.
- 남은 1904B 저장을 위해 FAT Entry를 다시 검색한다.
- 191번 클러스터를 할당한다.
### FAT 테이블 작성
- FAT 테이블은 다음과 같이 작성된다.
```text
FAT[190] = 191
FAT[191] = 0x0FFFFFFF
```
- `FAT[190] = 191`은 190번 클러스터 다음이 191번 클러스터라는 의미이다.
- `FAT[191] = 0x0FFFFFFF`는 191번 클러스터가 파일의 끝이라는 의미이다.
- Directory Entry에는 시작 클러스터 번호인 190이 기록된다.
### 실제 저장 후 슬랙 생성
- 사용 클러스터는 190번과 191번이다.
- 슬랙 공간이 발생한다.
- RAM Slack은 144B이다.
- File Slack은 2048B이다.

# 2차시: 삭제/손상 파일 복구 실습
## FAT 파일 삭제
### FAT 파일 삭제 동작 원리
#### 삭제 연산의 특징
- FAT 파일시스템의 삭제 연산은 실제 데이터를 지우지 않는다.
- 삭제 시 변경되는 것은 딱 두 가지이다.
#### FAT 엔트리 초기화
- 파일이 사용하던 클러스터의 FAT 엔트리를 초기화한다.
- 예시 클러스터는 190번, 191번이다.
- 해당 FAT 엔트리를 `0x00`으로 변경한다.
#### Directory Entry 수정
- 디렉토리 엔트리의 첫 번째 바이트를 수정한다.
- 첫 번째 바이트는 파일 이름의 첫 글자이다.
- **파일 이름**의 첫 글자를 `0xE5`로 변경한다. 
#### Data Area 유지
- Data Area에 저장된 파일 데이터는 그대로 유지된다.
- 삭제 연산은 파일 데이터 자체를 직접 지우지 않는다.
## 삭제된 FAT 파일 복구
### 삭제된 파일이 복구 가능한 이유
#### Data Area가 유지되는 구조
- 삭제 연산 시 데이터 영역인 Data Area를 전혀 건드리지 않는다.
- 파일의 실제 데이터는 삭제 직후에도 그대로 남아 있을 수 있다.
#### Directory Entry의 정보 유지
- Directory Entry는 첫 글자만 `E5`로 변경된다.
- 파일 크기는 그대로 유지된다.
- 시간 정보는 그대로 유지된다.
- 속성(Attribute)은 그대로 유지된다.
- 시작 클러스터 번호는 그대로 유지된다.
#### 삭제 파일 식별 방식
- 포렌식 도구에서는 `E5` 값을 기준으로 삭제된 파일을 식별한다.
#### 연속 클러스터 저장 시 복구 가능성
- FAT 엔트리가 0으로 초기화되었더라도 파일이 연속된 클러스터에 저장되어 있었다면 복구가 가능하다.
- 시작 클러스터와 파일 크기 정보만으로 완전 복구가 가능하다.
### 메타데이터가 모두 있는 경우의 복구
#### 이상적 시나리오
- Directory Entry에서 `E5`로 표시된 삭제 엔트리를 탐색한다.
- 확인 가능한 정보는 다음과 같다.
    - 파일 시작 클러스터
    - 파일 크기
    - 속성
    - 시간 정보
- Data Area에서 시작 클러스터부터 파일 크기만큼 데이터를 읽는다.
- 읽은 데이터를 통해 원본 파일을 복원한다.
- 이 방식은 가장 빠르고 정확한 복구 방식이다.
### 메타데이터가 없는 경우의 복구
#### 퍼즐 조각 방식
- Directory Entry 정보가 없는 경우이다.
- FAT 정보가 없는 경우이다.
- Data Area에 남아 있는 파일 조각인 클러스터를 분석한다.
- 파일 조각의 패턴을 분석한다.
- 구조적 특징을 분석한다.
- 파일 시그니처를 기반으로 다음 클러스터를 추정하며 재구성한다.
#### 특징
- 복구 시간이 증가한다.
- 정확도가 저하된다.
- 조각난 파일일수록 복구 난이도가 급격히 상승한다.
- 조각난 파일은 fragmented file이다.
### 연속 클러스터 저장 여부가 복구 난이도 결정
#### 연속적 저장
- 시작 클러스터와 크기 정보로 복구가 용이하다.
#### 조각난 저장
- 퍼즐 맞추기 방식이 필요하다.
- 복구 시간이 증가한다.
- 복구 결과 검증도 어렵다.
### 삭제 후 덮어쓰기 발생 시
- 삭제된 클러스터 영역인 Data Area에 새로운 파일이 기록될 수 있다.
- 새로운 파일이 기록되면 기존 데이터가 덮어써진다.
- 이 경우 원본 데이터가 소실된다.
- 원본 데이터가 소실되면 복구가 불가능하다.

# 3차시: NTFS 파일시스템
## NTFS 파일 시스템 내 파일 생성 및 삭제
### NTFS 등장 배경
#### FAT 파일시스템의 한계
- 파일 조각화 문제가 있다.
- 파일 조각화는 Fragmentation이다.
- 대용량 파일 처리에 비효율적이다.
- 안정성 기능이 부족하다.
- 보안 기능이 부족하다.
#### Microsoft의 대응
- Microsoft는 미래 대비형 파일시스템을 설계했다.
- NTFS는 1993년 Windows NT에서 최초 도입되었다.
- 현재 Windows OS의 주력 파일시스템이다.
- 서버 환경을 고려해 설계된 것이 핵심 차이이다.
### NTFS 주요 특징
#### 안정성 강화
- USN Journal 기반 변경 기록을 사용한다.
- 정전이나 강제 종료 시 롤백이 가능하다.
#### ADS
- ADS는 `Alternate Data Stream`이다.
- 하나의 파일에 여러 데이터 스트림을 저장할 수 있다.
#### 저장 효율
- 대부분 0인 데이터는 실제 저장을 생략할 수 있다.
- 파일 압축이 가능하다.
#### 보안
- EFS를 지원한다.
- EFS는 파일 단위 암호화이다.
- BitLocker와 연계 가능하다.
#### 백업
- VSS를 지원한다.
- VSS는 Shadow Copy이다.
- 이전 버전을 유지할 수 있다.
#### 설계 방향
- 전체적으로 안정성과 보안성 중심으로 설계되었다.
### NTFS의 기본 구조
#### VBR
- VBR은 `Volume Boot Record`이다.
- 부트 코드를 포함한다.
- BPB를 포함한다.
- BPB는 `BIOS Parameter Block`이다.

| 바이트 offset, 10진수 | 바이트 offset, 16진수 | 설명                          |
| ---------------- | ---------------- | --------------------------- |
| 0 - 2            | 0x0000 - 0x0002  | Jump command to boot code   |
| 3 - 10           | 0x0003 - 0x000A  | OEM ID("NTFS" 키워드 저장)       |
| 11 - 83          | 0x000B - 0x0053  | BIOS Parameter Block(BPB)   |
| 84 - 509         | 0x0054 - 0x01FD  | Boot code and Error Message |
| 510 - 511        | 0x01FE - 0x01FF  | Signature(0x55AA)           |

#### MFT
- MFT는 `Master File Table`이다.
- 파일·디렉토리 메타데이터 관리의 핵심이다.
- 0번 엔트리는 `$MFT` 자체 정보이다.
- 엔트리는 디스크 전체에 흩어질 수 있다.
#### 데이터 영역
- 파일의 실제 내용을 저장한다.
#### FAT과의 관계
- FAT의 테이블 역할을 MFT가 수행한다.
### NTFS 접근 흐름
- VBR에 접근한다.
- BPB에서 MFT 시작 위치를 확인한다.
- `$MFT`에 접근한다.
- `$DATA` 속성에서 전체 MFT 구조를 파악한다.
- 이후 파일과 디렉토리를 탐색한다.
- FAT의 `루트부터 따라가기` 개념과 논리적으로 동일하다.
### MFT 엔트리 구성 요소
#### 헤더
- Header 영역이다.
#### Fixup 영역
- 무결성 점검 및 복구 목적의 영역이다.
#### 속성 영역
- Attribute 영역이다.
- 주요 속성은 다음과 같다.
    - `STANDARD_INFORMATION`
    - `FILE_NAME`
    - `DATA`
#### 파일과 MFT 엔트리 관계
- 하나의 파일은 하나의 MFT 엔트리로 관리된다.
### 파일 생성 과정
#### 예시 조건
- 파일명은 `file1.dat`이다.
- 파일 크기는 4000B이다.
- 클러스터 크기는 2048B이다.
- 따라서 클러스터 2개가 필요하다.
#### 생성 흐름
- VBR에서 MFT 시작 위치를 확인한다.
- `$MFT`의 `$BITMAP`에서 미사용 엔트리를 탐색한다.
- 해당 MFT 엔트리를 초기화한다.
- 파일 속성을 기록한다.
- 기록하는 파일 속성에는 `FILE_NAME`, 크기 등이 포함된다.
- 할당 알고리즘에 따라 빈 클러스터를 배정한다.
- 디렉토리 인덱스 엔트리를 생성한다.
- 모든 변경 사항을 `LogFile`에 기록한다.
- 모든 변경 사항을 `UsnJrnl`에 기록한다.
- NTFS는 모든 변화가 기록되는 구조이다.
## 파일 카빙
### 삭제 복구 개념
#### FAT과 동일한 핵심 철학
- FAT과 동일하게 데이터는 남고 인덱스만 변경된다.
#### $BITMAP 기반 비사용 영역 분석
- `$BITMAP`을 통해 비사용 영역을 분석한다.
#### MFT 엔트리 기반 데이터 영역 재획득
- MFT 엔트리의 `FILE_NAME`을 이용한다.
- 파일 크기 정보를 이용한다.
- MFT 엔트리 정보를 이용해 데이터 영역을 재획득한다.
#### 메타데이터가 없는 경우
- 메타데이터가 없는 경우 비할당 클러스터 분석이 가능하다.
#### FAT과 NTFS의 차이
- FAT과 유사하지만 관리 단위는 MFT이다.
### 포렌식 분석 기법
#### 슬랙 공간 분석
- 파일 슬랙을 분석한다.
- 파일시스템 슬랙을 분석한다.
- 볼륨 슬랙을 분석한다.
- 슬랙 발생 원인을 분석한다.
- 실제 데이터 크기와 관리 단위 크기가 다르기 때문에 슬랙이 발생한다.
#### 파일 카빙
- 파일 카빙은 File Carving이다.
- 메타데이터 없이 파일을 복구하는 기법이다.
- 시그니처 기반으로 복구한다.
- 시그니처는 헤더와 푸터를 의미한다.
- 구조가 사라져도 퍼즐처럼 복원할 수 있다.
- JPG, GIF 등 고유 패턴을 활용한다.
