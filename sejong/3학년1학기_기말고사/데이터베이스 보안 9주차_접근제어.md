# 1장. 접근제어(Access Control) 개요
## 1.1 접근제어 개념
### 1.1.1 분야별 정의
* 정보시스템: 사용자, 프로그램, 프로세스 등이 시스템 자원에 접근하는 것을 제한하는 보안 기술
* 통신망: 통신 장비가 전송 매체에 접근하는 것을 제어하는 것
* 데이터베이스: 사용자가 DBMS에 로그인 하거나 SQL을 수행하려고 할 때 미리 정의된 **보안규칙에 따라 권한 여부를 판단하여 통제**하는 솔루션 -> SQL 통제 및 로깅이 필요한 SQL에 대하여 SQL 수행과 관련된 정보를 저장하는 기능 보유
## 1.2 접근제어 수행 요소
### 1.2.1 주체(Subject: S)
* 사용자, 프로그램, 프로세스 등 객체에 접근을 시도하는 대상
	* 소유자: 자원의 생성자, 시스템의 관리자 등
	* 그룹: 특정 그룹에 속한 사용자는 그룹의 접근 권한을 동일하게 유지
	* 전체: 소유자, 그룹에 속하지 않은 사용자(Guest)로, **최소한**의 접근 권한 부여
### 1.2.2 객체(Object: O)
* 데이터베이스, 서버, 응용 프로그램 등 주체가 접근하고자 하는 대상
### 1.2.3 접근 권한(Access rights/Privilege/Authority)
* 특정 주체가 특정 객체를 접근하는 방법을 지정
	* 읽기(Read, R): 주체는 객체의 정보를 볼 수 있고 복사(Copy) 또한 가능
	* 쓰기(Write, W): 주체는 객체에 대한 수정, 추가 및 삭제 가능
	* 실행(Execute, X): 주체는 객체 실행 가능
	* 삭제(Delete, D): //
	* 생성(Create: C): //
## 1.3 접근제어 절차
* **식별 → 인증 → 인가**의 순서로 이루어짐
### 1.3.1 식별(Identification)
* 시스템이 S의 식별자를 확인하는 과정
* 개인 식별자는 유일해야함
### 1.3.2 인증(Authentication)
* 사용자 인증: 정보에 접근할 수 있는 S의 자격 검증
* 메시지 인증: 전송된 메시지가 송신자가 보낸 그대로의 것인지 확인
- 패스워드, 토큰, 지문 등으로 인증함.
### 1.3.3 인가(Authorization)
* 주체가 객체에 대한 사용 권한을 부여 받는 것.
# 2장. 데이터베이스 접근제어 설계
* 접근제어 규칙이 잘못 설정되면 악의적인 사용자가 중요 정보에 접근하여 유출하거나 변조할 수 있음
* 따라서 정확하고 정밀한 접근제어 규칙 설계가 필요함
## 2.1 접근제어 규칙 정의
접근제어 규칙은 다음 4가지 요소로 구성됨
### 2.1.1 정보 식별
- 개인정보와 같은 접근제어 할 대상 정보 식별
- 조직의 성격에 따라 분류하여 특별히 관리 필요
### 2.1.2 사용자 식별
- 관리 대상 정보에 접근할 수 있는 사용자를 파악하는 것.
- 사용자 ID, IP 및 MAC 주소 등을 이용해 식별
### 2.1.3 접근 방법
- 정보에 접근할 수 있는 방법 통제
- DB접속에 사용하는 접속 툴, 접속 프로토콜 지정
### 2.1.4 접근 시간
- 정보에 접근할 수 있는 시간 통제
* 업무 시간, 비업무 시간, 주말, 공휴일 등 기준 설정 가능
## 2.2 접근제어 규칙 종류 및 설정
### 2.2.1 명칭부
* 규칙의 명칭을 정의해 규칙의 내용을 간단하게 표현할 수 있음
### 2.2.2 조건부
* 규칙 적용 조건을 설정
* 다양한 Factor의 결합으로 구성됨
### 2.2.3 제어부
* 사용자가 접속을 했을 때 조건을 만족할 경우 수행할 보안 제어 동작 설정
	* 로그인 통제
	* SQL 수행 통제
	* 로깅 통제
	* 경보 통제
#### 2.2.3.1 제어부: 로그인 통제 규칙
- DBMS에 로그인하는 과정을 통제하는 규칙
- Multi-Factor를 조합하여 DBMS에 대한 접속 허용 여부를 결정
#### 2.2.3.2 제어부: SQL 수행 통제 규칙
- SQL을 수행할 때마다, 사용자의 SQL 수행 권한을 확인해 실행 허용 여부를 결정하는 규칙
#### 2.2.3.3 제어부: 로깅 통제 규칙
- 저장되는 SQL 정보를 설정하는 규칙
- 불필요한 SQL을 저장하지 않도록 설정을 하는 것이 중요!
#### 2.2.3.4 제어부: 경보 통제 규칙
* 보안 정책 위반하는 사건 발생시 보안 관리자에게 통지하여 대응을 할 수 있게하는 규칙
	* 반복적인 인증 실패, 접근 금지된 객체에 대한 접근 시도
* 이메일, SMS 등으로 경보 메시지 전송 가능
# 3장. 데이터베이스 접근제어 구축
접근제어 설계 이후 접근제어 구축
## 3.1 우회 접근 방지
**주기적 점검 및 우회경로 즉시 차단**
### 3.1.1 개요
* DB 우회 접근은 우회 경로를 통해 접근제어를 우회하는 것을 의미하고 이에 대한 차단 필요
- 접근제어 결과가 기록될 수 있도록 로깅 설정 필요 => 책임추적성(Accountability) 보장
* 프록시 게이트웨이(Proxy Gateway)
	* 접근제어 서버를 구축하는 것
	* DB 접속 시 프록시 서버를 경유하는 걸 강제하여 접근제어 수행
### 3.1.2 에이전트 설치에 의한 차단
* DB 서버에 에이전트를 설치하여 접속 제어 수행
* DB 서버에 설치하므로 기존 서버에 대한 영향을 최소화하는 것이 필요
### 3.1.3 서버 보안에 의한 차단
- 운영체제 커널 단에 프로그램을 설치하여 특정 API 호출 시 권한을 점검하고 실행하는 솔루션
- DBMS가 설치된 서버에 설치하여 특정 프로그램이나 특정 포트에 대한 접근제어 수행
### 3.1.4 스니핑 방식에 의한 차단
* 패킷을 복사하여 트래픽을 감시하고 우회 접속을 탐지 및 차단
- DB 서버에 영향 X
### 3.1.5 기타 차단 방법
* 방화벽, 네트워크 분리, 로그인 트리거 등을 활용
- DB 서버로 직접 접근할 수 있는 경로를 최대한 제어
## 3.2 환경 보안
### 3.2.1 접근제어 서버 관리
- 접근제어 서버가 안전하지 않으면 무의미하므로 DBMS와 동일한 수준으로관리가 필요.
### 3.2.2 DB 접근제어 관리자 계정 관리
- 관리자에게 적절한 관리 권한이 부여되도록 보안 정책 설정
### 3.2.3 포트 관리
* 사용되지 않는 포트는 닫고, 필요한 포트만 최소 개방
### 3.2.4 OS 취약점 관리
- 정기적으로 OS 취약점 점검 수행
* 운영체제의 취약점을 최신 패치로 보완
## 3.3 보안 적용 시험
### 3.3.1 개요
* 접근제어 솔루션이 제대로 작동하는지 점검
- 개발 DB가 별도로 구축된 상태에서 진행
### 3.3.2 시험 대상 항목
* 로그인 제어, SQL 제어, 로깅, 경보 등
### 3.3.3 시험 환경 구성
- 실제 환경 또는 시험 항목에 따라 다르게 구성 가능
### 3.3.4 보안 적용 시험 수행
- 기능 시험: 필요한 기능이 적절히 동작하는지 확인
### 3.3.5 유의점
- 기능 시험에서 조직이 DB 접근제어를 위해 필요하다고 생각하는 모든 기능 점검
# 4장. 임의적 접근 제어(DAC) 개요
**접근제어 모델(정책)**
주체가 어떻게 객체에 접근하는지를 규정하는 방법
## 4.1 임의적 접근제어(DAC)
- Discretionary Access Control
- 접근을 요청하는 사용자의 신분과 접근 규칙에 기반하는 접근제어
- 객체 소유자의 판단에 따라 임의적으로 권한을 줄 수 있다.
### 4.1.1 장점
- 융통성이 있는 시스템
- 구현이 용이하고 사용이 간단
- 객체별로 세분환된 접근제어 가능
### 4.1.2 단점
* 권한이 전이될 수 있어 다른 모델에 비해 보안에 취약
* 정보에 대한 엄격한 접근제어 어려움
### 4.1.3 예시
- Microsoft의 Windows 운영체제처럼 대부분의 OS에서 DAC 기반 정책을 구현함
# 5장. 임의적 접근 제어(DAC) 모델
## 5.1 임의적 접근제어(DAC) 모델
### 5.1.1 대표적인 모델
* 접근제어행렬(Access Control Matrix, ACM)
* 권한목록(Capability List, 접근가능목록)
* 접근제어목록(Access Control List, ACL)
### 5.1.2 접근제어행렬(ACM)
- 모든 주체와 객체에 대한 권한 관계를 2차원 배열로 표현
### 5.1.3 권한목록(Capability List)
- 주체별로 접근 가능한 객체를 리스트 형태(객체, 권한)로 구현
- 주체별 객체가 많아지면 탐색 시간 증가
- 토큰: 객체에 대한 권한 정보 보유
### 5.1.4 접근제어목록(ACL)
- 객체별로 객체에 접근 가능한 주체를 리스트 형태(주체, 권한)로 구현
- 객체별 권한을 가진 주체가 많아지면 탐색 시간 증가
### 5.1.5 Trojan Horse 공격에 취약
- 응용프로그램에 내장된 Trojan Horse에 대해 데이터 보호 불가
* 권한을 위임받은 사용자 계정을 악용하여 악성코드가 접근을 시도할 수 있음
