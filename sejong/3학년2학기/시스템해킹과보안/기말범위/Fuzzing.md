# 1.  개요
- '퍼즈 테스팅'이라고도 하며, **소프트웨어 대상**에게 무작위 또는 반 무작위 값을 입력하여 상태를 모니터링하여 버그를 탐지하는 목적.
- 탐지된 버그는 취약점으로 이어질 수 있음 -> 시스템 권한을 획득
# 2. 퍼징의 분류
## 2.1 필요정보에 따른 분류
### 2.1.1 블랙박스 (Sulley, Boofuzz, Peach)
- 무작위 또는 반무작위 값을 대상에 입력하고 모니터링하는 방식
- **brute-force 취약점 탐지기**라고도 불림
- 장점: Easy to Use, 사전 작업 필요 X
- 단점: 낮은 코드 커버리지, 무의미한 시도 많음
### 2.1.2 화이트박스 (Driller, QSYM, T-Fuzz)
- 대상의 내부 구조 정보와 실행 정보 등 **프로그램 전반에 대한 정보**를 바탕으로 퍼징 수행 ex) Symbolic Execution
- 장점: 코드 커버리지가 넓고 효과적임
- 단점: 사전 정보 수집이 필요, 제약 조건 해결기 필요, 제약 조건 해결을 위한 **state explosion 문제** 
### 2.1.3 그레이박스 퍼징 (AFL, libFuzzer, VUzzer)
- 블랙박스와 화이트박스의 중간 접근법. **대상 내부의 일부 정보**를 이용해 퍼징을 수행
- 화이트 박스 퍼징과 다르게 **대상 내부의 전체 의미를 추론하지 않음** -> 가벼운 정적 분석.
- 장점: 중간 접근법이므로 구현에 따라 **효율적인 퍼징**이 가능 than 블랙박스
- 단점: **오버헤드가 많음** than 블랙박스
## 2.2 테스트케이스 생성 방법에 따른 분류
### 2.2.1 Generation-based Fuzzing (Smart Fuzzing)
- 퍼징을 수행할 대상에 대한 **포맷이나 명세를 해석함**으로써 잘 형성된 입력 값 생성
- 장점: 
	- **정의된 규칙**에 따라 입력값이 생성되므로 안정성이 있음
	- 대상에 Checksum과 같은 **복잡한 의존성**이 있는 경우에도 해결 가능
- 단점: 
	- 명세를 작성하기 위한 **분석시간이 필요**
	- 사용자가 작성한 **명세의 품질**에 따라 퍼징 성능에 대한 의존성이 있음
### 2.2.2 Mutation-based Fuzzing
- **초기 입력 값(seed)를 변형**하는 방식으로 계속적인 테스트케이스를 생성하는 방식 
	- ex) bit-flipping, arithmetic inc/dec, random bytes
- 장점: **퍼징 환경 설정**과 **자동화**가 용이
- 단점: **초기 입력 값이 제한적**이면 대상의 Checksum이나 다른 검사들로 인해 **깊은 탐색이 어려워**질 수 있음
## 2.3 대상에 따른 분류
- 네트워크 퍼징, 파일 퍼징, 커널 퍼징 등
# 3. 퍼징 수행 6단계
## 3.1. 인터페이스 식별 (Identifying Interfaces)
외부 입력을 받아들이는 **모든 인터페이스**를 식별해야 함.
- **식별 대상:**
    - 네트워크 소켓 (프로토콜, 포트 번호 등으로 구성됨)
    - 포맷이 있는 파일들 (Formatted Files)
- **주의점:**
    - 때로는 인터페이스가 명확하지 않을 수 있음.
    - 예: 웹 브라우저의 경우 HTTP, FTP, RTSP, 이미지 포맷 등 다양한 인터페이스가 존재함.
- **효과:** 인터페이스를 잘 식별하는 것은 테스팅 커버리지(Testing Coverage)를 높이는 데 기여함.
## 3.2. 입력 생성 (Input Generation)
- 퍼징된 입력을 생성하는 것은 **퍼저(Fuzzer)의 심장이자 영혼**과도 같음. 프로토콜이나 구조에 대한 지식이 거의 없는 상태부터 완벽히 이해하는 상태까지 다양한 생성 방법이 존재함.
### 3.2.1. 가장 기본적인 방법: 순수 무작위 데이터 (Random Data)
- 단순히 순수한 무작위 데이터를 공급하는 방식임.
- 이론적으로는 프로그램에 가능한 모든 입력을 다 넣어볼 수 있음.
- 하지만 현실적으로 **시간 제약(Time constraints)** 때문에 효율성이 떨어짐.
- 따라서 이 방법만으로는 유의미한 결과를 얻기 어려움.
### 3.2.2. 변이 기반 접근 (Mutation-based approach)
- 먼저 시스템에 대한 **유효한 입력** 을 수집함. (예: 네트워크 패킷 캡처, 유효한 파일들, 커맨드 라인 인자 등)
- 수집된 입력에 **변칙**을 추가하는 방식임.
- **변칙 추가 방법:**
    1. 구조를 무시하고 **무작위로** 추가함.
    2. 프로토콜에 대한 **내장된 지식**을 활용하여 좀 더 지능적으로 추가함.
    3. 일부 퍼저는 구조를 자동으로 **분석**하려고 시도하기도 함.
### 3.2.3. 생성 기반 퍼저 (Generation-based Fuzzers)
- 예: Peach, SPIKE, Sulley 등.
- 유효한 테스트 케이스가 필요 없음.
- 이미 기본 프로토콜이나 입력 포맷을 **이해하고 있음**.
- **특징:**
    - 입력 포맷에 대한 지식 수준에 따라 품질이 결정됨.
    - 문서나 포맷 명세(Specification)를 읽고, 이를 바탕으로 퍼징 입력을 생성하는 **프레임워크(Framework)**를 제공함.
- **단점:**
    - 명세(Specification)를 작성하는 데 전문 지식이 필요하며, 때로는 수백 시간이 소요됨.
    - 알려지지 않았거나 독점적인(Proprietary) 포맷을 테스트하려 할 때는 지원되지 않을 수 있음.
## 3.3. 타겟에 입력 전달 (Sending Inputs to the Target)
- 일부 퍼저는 이 과정을 사용자에게 맡기기도 하지만, 대부분의 퍼저는 입력 전달 방법을 제공함.
- **네트워크 프로토콜의 경우:** 타겟 서버에 반복적으로 **연결**을 시도하거나, 클라이언트 측 퍼징을 위해 포트를 열고 수신 대기함.
- **파일 포맷의 경우:** 퍼징된 파일과 함께 타겟 애플리케이션을 반복적으로 **실행**함.
## 3.4. 타겟 모니터링 (Target Monitoring)
퍼징 입력을 만들어 보내더라도, 그게 언제 문제를 일으켰는지 모르면 소용없음.
- **모니터링 방법:**
    - **기본적인 방법:** **디버거** 하에서 실행하고 감시함.
    - **정교한 방법:** 텔넷(telnet)이나 SSH 등을 통해 타겟 애플리케이션, 로그 파일, 시스템 리소스, 기타 파라미터 등을 **모니터링**함.
- **고급 모니터링의 장점:**
    - **분석 속도 향상:** 어떤 입력이 특정 에러 조건을 유발했는지 정확히 판단할 수 있게 해 줌.
- **자동 재시작 기능:**
    - 애플리케이션을 단순히 재시작하거나, 스냅샷으로부터 가상 머신(VM)을 복구하는 기능이 중요함.
- **핵심:** "모니터링이 고도화될수록, 더 많은 취약점을 발견하게 될 것임!"
## 3.5. 예외 분석 (Exception Analysis)
- 발견된 각 충돌(Crash)이나 에러 조건에 대해, 해당 결함을 재현할 수 있는 **최소한의 입력 시퀀스**를 찾아내야 함.
- 그 후, 모든 에러를 분류하여 **고유한 취약점**이 몇 개인지 파악해야 함.
    - 예: 하나의 '포맷 스트링 취약점'이 여러 다른 외부 함수를 통해 도달 가능할 수 있음. 즉, 많은 테스트 케이스가 크래시를 유발하더라도 실제로는 모두 같은 취약점을 가리킬 수 있음.
## 3.6. 보고 (Reporting)
- 개발팀과의 의사소통을 위해 필요함.
- 클라이언트에게 제출할 문서를 만들기 위해 필요함.
- 일부 퍼저는 유용한 통계, 그래프, 예제 코드, 에러를 재현할 수 있는 작은 바이너리 등을 제공하기도 함.
# 4. AFL (American Fuzzy Lop) 개요
- **그레이박스 퍼저**
- **커버리지 기반(Coverage-guided):** 변형된 입력 값이 새로운 코드 실행 경로(Path)를 발견하면, 해당 입력을 큐(Queue)에 저장하여 다음 변이의 재료로 씁니다. 이 과정을 통해 효율적으로 코드 커버리지를 넓혀갑니다
## 4.1 주요 구성 요소
- `afl-fuzz`: 실제 퍼징을 수행하는 핵심 도구.
- `afl-gcc`: 소스 코드를 컴파일할 때 사용하는 래퍼(Wrapper). 컴파일 과정에서 계측(Instrumentation) 코드를 삽입
- `afl-analyze`: 파일 포맷 분석기.
- `afl-cmin`, `afl-tmin`: 테스트 케이스 중복 제거 및 최소화 도구.
## 4.2 결과 모니터링 및 분석
- **UI 화면:** 실행 시 대시보드가 나타나며 `Process timing`, `Stage progress`, `Total crashes` 등의 정보를 보여줌.
- **결과 폴더:** 작업 디렉토리 내에 다음과 같은 폴더가 생성됨
    - `queue`: 코드 커버리지를 높인 유의미한 입력값들
    - `hangs`: 타임아웃(응답 없음)을 유발한 입력값들
    - **`crashes`**: 프로그램 충돌을 일으킨 입력값들 (**핵심 분석 대상**)
