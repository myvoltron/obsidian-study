# 1장. 강제적 접근제어(MAC) 개요
## 1.1 개념
- Mandatory Access Control(= Rule-based Access Control)
* 주체와 객체의 보안 분류 등급에 기반한 접근 제어 방식
- 미리 정해진 정책에 기반해 **주체에게 허용된 접근권한**과 **객체에 부여된 허용 등급**을 **비교**하여 접근제어 수행
## 1.2 특징
* 보안 정책과 등급은 **관리자만 변경 가능**
- 다른 접근 통제 모델에 비해 **접근 규칙의 수가 적고, 통제 용이**
## 1.3 장단점
**장점**
* 중앙 집중식 관리 → **DAC**보다 높은 보안성 제공
**단점**
* 모든 접근에 대해 보안 등급과 정책 정의 필요 → 구현 난이도↑
- 많은 관리적 부담 요구
* 상업 환경에 부적합 (유연성 부족)
## 1.4 예시
* **군 조직**: 계급에 따라 접근 가능한 군사 기밀 등급 결정
  * 예: 사병은 ‘기밀(Confidential)’ 문서까지만 접근 가능
* **온라인 카페**: 게시판1(골드 이상), 게시판2(플래티넘 이상)
# 2장. MAC 대표 모델
## 2.1 **Bell-LaPadula (BLP)** 모델 – 기밀성 중심
### 2.1.1 개요
* **정보의 기밀성 보호**에 중점을 둔 모델
- 주체와 객체의 보안 수준을 등급으로 구분해 MAC 적용
### 2.1.2 보안 속성
* **Simple Security Property, SS**: No Read Up
  → 주체의 등급 ≥ 객체의 등급이어야 읽기 가능
* **Star Security Property**: No Write Down
  → 주체의 등급 ≤ 객체의 등급이어야 쓰기 가능
### 2.1.3 DB 보안 예시

| 보안 수준        | 주체  | 객체   |
| ------------ | --- | ---- |
| Top Secret   | 교수  | 성적   |
| Secret       | 조교  | 출석표  |
| Confidential | 학생  | 강의자료 |

**읽기 권한**: No Read Up → 교수는 모든 객체 읽기 가능, 학생은 강의자료만 읽기 가능
**쓰기 권한**: No Write Down → 교수는 성적만 수정 가능, 조교는 출석표까지만 가능

=> 학교 성적 시스템 적용에 적합하지 않음!
### 2.1.4 한계점
* 기밀성은 보장되지만, **무결성(Integrity)은 보장되지 않음**
* 보안 수준 낮은 주체가 높은 객체를 **수정 가능** → 비정상적인 정보 조작 가능성 존재
=> Biba 모델 등장
## 2.2 Biba 모델 – 무결성 중심
### 2.2.1 개요
* BLP의 무결성 한계 보완 목적
* **정보의 무결성 유지**에 중점
### 2.2.2 무결성 속성
* **Simple Integrity Axiom**: No Read Down
  → 주체의 등급 ≤ 객체의 등급이어야 읽기 가능
* **Star Integrity Axiom**: No Write Up
  → 주체의 등급 ≥ 객체의 등급이어야 쓰기 가능
### 2.2.3 DB 보안 예시

| 무결성 수준       | 주체  | 객체   |
| ------------ | --- | ---- |
| Top Secret   | 교수  | 성적   |
| Secret       | 조교  | 출석표  |
| Confidential | 학생  | 강의자료 |
**읽기 권한**: 교수는 성적만 읽기 가능, 학생은 강의자료까지만 읽기 가능
**쓰기 권한**: 교수는 강의자료까지 쓰기 가능, 학생은 강의자료만 가능
### 2.2.4 한계점
* 기밀성이 보장되지 않음 → 학생이 상위 등급 객체를 읽을 수 있음
# 3장. 역할기반 접근제어 (RBAC)
## 3.1 개념
- Role-Based Access Control
- 주체에 대한 행위나 역할에 기반하는 접근제어
* 사용자에게 직접 권한을 부여하는 대신, **역할(Role)을 부여하고, 역할이 가진 권한을 통해 접근제어 수행**
## 3.2 기존 접근제어 모델과의 차이점
- 사용자 정보가 자주 바뀌는 경우
	- DAC, MAC -> 지속적인 권한 관리 필요
	- RBAC -> 역할 부여 및 취소 가능
- 예) 회원가입 및 탈퇴
	- 권한 관리의 어려움 -> 역할에 따라 권한을 다르게 부여
## 3.3 구성 요소
* **사용자(User)**: 자원에 접근하고자 하는 대상
* **자원(Resource)**: 무단 액세스로부터 조직이 보호해야하는 대상
* **권한(Permission)**: 객체에 대한 기능(읽기, 쓰기 등)
* **역할(Role)**
## 3.4 수행 과정
- 단순한 형태의 접근제어 수행
	1. 역할 생성
	2. 사용자 ↔ 역할 연결 (Mapping)
	3. 사용자의 접근 → 접근 제어 수행
- 핵심 개념
	- 사용자 할당(User Assignment, UA)
	- 권한 할당(Permission Assignment, PA)
## 3.5 등장 배경
- 기존의 접근제어: 사용자 중심 권한 부여 -> 빈번한 수정 발생 시 비효율적
- RBAC: 효율적 권한 관리 가능
## 3.6 장단점
**장점**
- 역할 회수 및 부여 -> 효율적(수행 작업 축소)
**단점**
* **역할 폭발(Role Explosion)**
	* 역할 수가 너무 많아 관리 어려움
	* 빈번한 역할 생성에 따라 역할 폭발 발생
## 3.7 RBAC 적용 분야
- 데이터베이스, 시스템 관리, 운영체제 제품
# 4장. RBAC 참조 모델 (NIST 기준)
## 4.1 기본형 RBAC(Flat/Core RBAC)
### 4.1.1 핵심 구성 요소
- 사용자(Users, U) - 사람이나 자율적인 에이전트
- 역할(Roles, R) - 조직과 관련된 직무, 책임, 자격
- 권한(Permissions, P) - 하나 이상의 객체들(Objects, OBS)에 대한 접근의 승인이나, 특정 행위(Operations, OPS)를 할 수 있는 특권
- 세션(Sessions, S) - 사용자의 특정 권한 사용을 위해 사용자와 역할 간의 세션 생성 및 역할(Role) 할당 -> 한 사용자를 여러 역할에 매핑 가능
	- 사용자와 역할의 M:N 매핑을 위한 테이블이라고 생각해도 될 듯 
### 4.1.2 구성 요소의 관계
- 사용자(U)는 역할(R)에 배정(User Assignment, UA)되고, 권한(P)도 역할(R)에 배정(Permission Assignment, PA)됨 
- U 특정 S 안에서 사용 가능한 모든 역할 내부의 권한들을 소유
- 사용자-역할, 권한-역할 배정은 다대다(Many- To-Many) 관계이고, 사용자는 동시에 여러 세션 소유 가능
## 4.2 계층형 RBAC(Hierarchical RBAC)
기본형 RBAC에 **역할 간의 계층 구조**를 추가한 형태
- 일반적 계층형 RBAC(General Hierarchical RBAC, GH-RBAC): **다중 상속 허용**
- 제한적 계층형 RBAC(Limited Hierarchical RBAC, LH-RBAC): 다중 상속 금지, **1단계 직속 상속만 허용**
## 4.3 제약형 RBAC(Constrained RBAC)
- 계층형 RBAC에 **임무분리**를 추가한 형태
- **임무분리(Separation of Duty(ies), SoD)**
	- 사용자 간의 이해관계 충돌을 막아 **사용자가 직위를 벗어나는 행위를 할 수 없게 하는 것**
	- 정적 임무분리 Static Separation of Duty(SSD)
		- 동일 사용자에게 정적인 임무분리가 선언된 **두 개 이상의 역할에 배정되지 못하게 하는 것**
	- 동적 임무분리 Dynamic Separation of Duty(DSD)
		- 동일 사용자에게 동적인 임무분리가 선언된 두 개 이상의 역할에 배정될 수 있지만, 동일 사용자가 동시에 이 역할들을 사용할 수 없게 하는 것
## 4.4 대칭형 RBAC(Symmetric RBAC)
- 제한형 RBAC에 **권한 배정(PA) 제약조건**을 추가한 형태
- **적절한 권한-역할 배정**은 권한부여 관리구조의 핵심적인 요소
- 과거의 권한-역할 배정은 새로운 상황 변화에 따라 적절하지 않을 수 있으므로 상황 변화에 능동적인 대처, 관리상의 다양한 상황 고려 필요

# 5장. 속성 기반 접근제어(ABAC) 모델 
## 5.1 개념
- Attribute-Based Access Control
* 동적인 환경과 S와 O의 속성(Attribute)에 기반한 접근제어
## 5.2 주요 요소
* **속성(Attribute)**: ABAC의 핵심 요소
	* 수치 또는 형태로 표현되는 유무형 정보
	* **환경 조건**: 주체나 객체와는 **독립적인 동적** 요소
* **주체(Subject)**: 특정 객체에 접근하고자 하는 대상
	* 사용자, 그룹, 애플리케이션
	* **주체 속성(Subject Attribute)**: 주체가 가진 속성들로 생년월일, 이름, 직업 등이 해당됨
* **객체(Object) = 자원(Resource)**: 무단 액세스로부터 보호해야하는 대상
	* **객체 속성(Object Attribute)**: 유형, 이름, 생성일자, 경로 등이 해당됨
## 5.3 동작 방식
1. 주체가 객체에 접근 요청
2. 속성 및 환경 정보를 기반으로 정책 판단
3. **속성들이 정책 만족** 시 접근 허용, 불일치 시 차단
## 5.4 장단점
**장점**
* **동적인 환경 조건**을 기반으로 세밀한 접근제어 ex) 특정 요일, 시간
* **객체와 주체 속성**을 통한 세밀한 접근제어 ex) 의료분야
* **유연한** 접근제어
**단점**
- 객체와 주체에 대한 속성의 **주기적, 개별적인 관리** 필요
- 인가 여부 결정 과정에서 **높은 성능 비용**이 발생, 그러나 감당할 수 있는 비용이라 큰 문제 X
## 5.5 활용
* **XACML(eXtensible Access Control Markup Language)**을 통해 다양한 프로그램에서 널리 사용
